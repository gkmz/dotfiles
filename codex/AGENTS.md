## 连续开发模式

当用户明确说“启用连续开发模式”“连续开发模式”“按连续开发模式做”“进入连续开发”时，进入本模式。普通开发、修 bug、写文档、代码评审或一次性小改动不自动启用本模式。

连续开发模式的目标是：把大功能拆成可独立完成、可独立检查的小功能点；每完成一个功能点就做一次质量检查；检查不通过先修复，不继续下一个功能点；所有功能点和整体回归都通过后，再提交代码。

### 启动流程

1. 先阅读项目说明、现有代码、测试方式、构建方式和相关文档。
2. 明确用户目标、非目标、约束条件和验收标准；如果需求存在高风险歧义，先问一个必要问题。
3. 将大功能拆成小功能点，每个功能点必须能独立实现、独立验证、独立回滚。
4. 给出开发顺序，并说明每个功能点的完成标准、预计影响文件、验证方式和主要风险。
5. 开始开发前，记录当前工作区状态，避免混入用户已有无关改动。

### 功能点开发循环

对每一个功能点，按以下循环执行：

1. 实现当前功能点，保持改动范围聚焦。
2. 更新必要的文档、文案、配置、迁移或注释。
3. 运行当前功能点相关的最小验证，例如单元测试、构建、类型检查、lint、手工命令或 UI 验证。
4. 执行“功能点检查清单”。
5. 如果检查不通过，先修复当前功能点，再重新检查。
6. 检查通过后，向用户简要说明当前功能点完成情况和验证结果，再继续下一个功能点。

不要在功能点之间提交代码。连续开发模式下，提交只发生在全部功能完成、全部检查通过、整体回归通过之后。

### 功能点检查清单

每完成一个功能点，必须检查：

- 产品/设计：行为是否符合用户目标，交互路径是否完整，文案是否自然一致，默认值是否合理。
- 代码质量：命名是否准确，职责是否清晰，结构是否贴合现有代码风格，是否引入不必要抽象或重复逻辑。
- 文档注释：README、变更日志、配置说明、用户文案或开发说明是否需要更新；复杂逻辑是否有必要的简短注释。
- 测试验证：是否新增或更新测试；是否运行相关测试；如果不能运行，说明原因和剩余风险。
- 边界问题：空状态、旧数据迁移、异常输入、重复点击、暂停/恢复、失败路径、并发/异步时序、用户取消操作。
- 安全问题：权限、文件读写、命令执行、网络请求、敏感信息、注入风险、越权访问。
- 兼容/回归：是否破坏已有快捷键、配置、构建脚本、公开 API、UI 布局、老用户数据。
- 可运维性：错误信息是否可理解，日志或状态反馈是否足够，失败时是否能恢复。

### 整体回归检查

全部功能点完成后，必须执行整体回归：

1. 查看 `git diff`，确认改动范围和需求一致。
2. 查看 `git status`，确认没有意外文件、生成物或用户无关改动被混入。
3. 运行项目可承受范围内最完整的验证，例如全量测试、构建、类型检查、lint 或关键手工流程。
4. 复查跨功能点交互，确认各功能点组合后没有状态冲突、重复逻辑或体验断裂。
5. 复查文档和注释，确认用户能理解新增行为，开发者能理解关键实现。
6. 复查安全和边界风险，明确无法覆盖的残余风险。

### 提交规则

只有在以下条件全部满足后，才可以提交代码：

- 所有功能点检查通过。
- 整体回归检查通过。
- 用户没有要求暂停、只看方案或暂不提交。
- `git diff` 和 `git status` 已复查。
- 没有把用户已有的无关改动纳入提交。

提交时：

- 优先使用一个聚合提交，commit message 描述完整功能，而不是中间步骤。
- 如果功能非常大且用户要求拆提交，按已完成并验证过的功能点拆分。
- 提交后在最终回复中说明提交哈希、验证命令和未覆盖风险。

### 中断与恢复

如果连续开发过程中被用户打断或需求变化：

1. 立即以最新用户消息为准。
2. 简要说明当前完成到哪个功能点、哪些检查已通过、哪些尚未开始。
3. 如果继续开发，重新确认剩余功能点顺序和检查标准。
4. 不丢弃用户或其他进程产生的改动，除非用户明确要求。

如果遇到无法继续的阻塞：

- 先尝试自行定位并提出最小可行替代方案。
- 如果需要用户决策，只问必要问题。
- 明确列出已完成内容、阻塞原因、风险和下一步建议。
